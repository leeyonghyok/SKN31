# 1. VectorRAG 한계와 GraphRAG의 필요성

## 1.1 RAG란?

RAG(Retrieval-Augmented Generation)는 LLM이 답변을 생성하기 전에 외부 데이터를 검색해서 참고하도록 하는 방식이다.

LLM만 사용할 때의 문제점:
- 학습 데이터의 cut-off 이후 정보를 모른다.
- 사내 문서, 개인 데이터 같은 비공개 정보에 접근할 수 없다.
- 환각(hallucination) 문제가 있다.

RAG는 이 문제를 외부 데이터 검색으로 보완한다. 질문에 관련된 문서를 먼저 찾고, 그 내용을 LLM에 함께 전달해서 답변을 생성한다.


## 1.2 Vector RAG의 기본 구조

Vector RAG는 RAG의 가장 기본적인 형태로, 벡터 유사도 검색을 사용해 관련 문서를 찾는다.

**저장 단계 (인덱싱)**

1. 문서를 일정한 크기의 청크(chunk)로 나눈다.
2. 각 청크를 임베딩 모델로 벡터로 변환한다.
3. 벡터를 VectorDB(예: Qdrant, Chroma, Pinecone 등)에 저장한다.

**질의 단계 (검색 + 생성)**

1. 사용자 질문을 벡터로 변환한다.
2. VectorDB에서 코사인 유사도 등으로 가장 가까운 벡터(청크)를 K개 찾는다.
3. 검색된 청크를 LLM 프롬프트에 포함해 답변을 생성한다.

```
정보저장
문서 → 청크 분할 → 임베딩 → VectorDB 저장

답변생성                                  
질문 → 임베딩 → 유사도 검색 → 상위 K개 청크(+질문)
                                  ↓
                                 LLM → 답변 생성
```


## 1.3 Vector RAG의 한계

Vector RAG는 **질문과 내용이 유사한 문서**는 잘 찾지만, 다음 유형의 질문에서는 한계를 보인다.

**1. 관계 기반 질문에 취약하다**

> "A 프로젝트에 참여한 직원들의 소속 부서는?"

이 질문은 직원 → 프로젝트 → 부서라는 관계를 따라가야 답할 수 있다. 그런데 "A 프로젝트"에 대한 청크와 "부서"에 대한 청크는 벡터 유사도가 낮을 수 있어서, 각각 따로 검색하면 연결 고리를 잃는다.

**2. 다단계 추론이 어렵다**

> "B가 만든 제품을 구매한 고객 중 재구매율이 높은 고객은?"

이 질문은 `제조사-[생산]-> 제품 <-[구매]-고객-[재구매]->제품` 와 같은 여러 단계의 연결을 거쳐야 한다. 청크 단위 검색은 이런 흐름을 따라가기 어렵다.

**3. 큰 범위의 내용을 바탕으로 하는 작업에 약하다**

> "이 회사의 전체 제품 라인업을 요약해줘."

보통 정보가 여러 청크에 분산되어 저장 되 있기 때문에, 유사도 검색만으로는 전체 범위를 커버하기 어렵다. 어떤 청크가 검색되냐에 따라 요약의 완성도가 달라진다.

**4. 희소하게 등장하는 정보를 놓친다**

유사도 기반 검색은 질문 문장과 표현이 유사한 청크를 선호한다. 관련 정보가 다른 용어나 맥락으로 기술된 경우 검색에서 누락될 수 있다.


## 1.4 GraphRAG가 필요한 상황

- GraphRAG는 데이터 안의 엔티티 간의 **관계**를 바탕으로 데이터를 검색한다. 그래서 위에서 언급한 Vector RAG의 한계를 보완 할 수 있다.

| 상황 | 예시 |
|------|------|
| 엔티티 간 관계 추적 | "A가 B를 통해 C와 어떻게 연결되는가?" |
| 다단계 연결 탐색 | "이 약의 부작용이 있는 성분을 포함한 다른 약은?" |
| 전체 구조 파악 | "이 논문이 인용한 논문들의 공통 저자는?" |
| 특정 조건의 경로 탐색 | "서울에서 부산까지 환승 횟수 2회 이하인 경로는?" |

이런 질문들은 **데이터가 노드와 노드간의 관계로 구성된 그래프 형태**일 때 자연스럽게 해결된다. 
- **Vector RAG가 "문서의 내용이 얼마나 유사한가"를 보는 반면, GraphRAG는 "엔티티가 어떻게 연결되어 있는가"를 본다.**

정리하면, 아래 조건 중 하나라도 해당하면 GraphRAG를 고려할 만하다:
- 데이터 안에 명확한 개체(사람, 장소, 제품 등)와 그 관계가 존재한다.
- 질문이 여러 엔티티 간 관계를 따라가야 답할 수 있다.
- 단순 키워드 매칭이나 의미 유사도 검색으로는 답이 불완전하다.
