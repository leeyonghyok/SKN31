# 3. GraphRAG 개요

## 3.1 GraphRAG란?

GraphRAG는 **사용자 질문과 관련된 정보**를 **지식 그래프에서 찾아**온 다음, 그 정보를 바탕으로 **LLM이 답을 생성하는 RAG 방식**이다.

Vector RAG는 텍스트 청크의 의미적 유사도를 기반으로 검색한다.  
그에 비해 **GraphRAG는 엔티티(노드)와 엔티티 간 관계(엣지)로 구성된 그래프를 탐색해서 관련 정보를 찾는다.**

핵심 아이디어:
- 문서에서 엔티티와 관계를 추출해 그래프로 저장한다.
- 질문이 들어오면 그래프를 탐색해서 관련 노드와 관계를 찾는다.
- 찾은 정보를 LLM에 전달해 답변을 생성한다.

## 3.2 GraphRAG의 전체 흐름

GraphRAG도 Vector RAG와 마찮가지로 **인덱싱**과 **질의** 두 단계로 나뉜다.

### 인덱싱 단계 (그래프 구축)

```text
원본 문서
    ↓
엔티티·관계 추출 (LLM 또는 NER(개체명 인식) 모델 사용)
    ↓
그래프 DB (예: Neo4j)에 노드(엔티티)·엣지(관계)로 저장
```

- 문서에서 "사람", "조직", "제품" 같은 엔티티를 추출한다.
- 엔티티 사이의 관계("소속", "제조", "구매")도 함께 추출한다.
- 추출한 구조를 그래프 DB에 저장한다.

### 질의 단계 (검색 + 생성)

```text
사용자 질문
    ↓
질문에서 핵심 엔티티 파악
    ↓
그래프 DB에서 관련 노드·관계 탐색 (Cypher 쿼리)
    ↓
탐색 결과를 LLM에 전달
    ↓
답변 생성
```

LangChain을 사용할 경우, `GraphCypherQAChain`를 이용하면 자연어 질문을 Cypher 쿼리로 변환하고 Neo4j에서 결과를 가져온 뒤 LLM으로 답변을 생성하는 전 과정을 처리할 수 있다.

## 3.3 GraphRAG와 Vector RAG의 차이

| 구분 | Vector RAG | GraphRAG |
|------|-----------|----------|
| 저장 형태 | 텍스트 청크 + 벡터 | 노드 + 엣지 (그래프) |
| 검색 방식 | 코사인 유사도 | 그래프 탐색 (패턴 매칭) |
| 강점 | 의미 유사성 기반 검색 | 관계·연결 기반 탐색 |
| 약점 | 관계 추론, 다단계 연결 | 구축 비용, 스키마 설계 |
| 쿼리 언어 | 없음 (유사도만) | Cypher (Neo4j 기준) |
| 적합한 데이터 | 비정형 텍스트 | 구조화된 관계 데이터 |

두 방식은 경쟁 관계가 아니다. 실제로는 Vector RAG와 GraphRAG를 함께 사용하는 하이브리드 방식이 더 강력한 경우가 많다.

## 3.4 GraphRAG를 사용하기 좋은 경우와 그렇지 않은 경우

GraphRAG는 단순히 “비슷한 문서 찾아줘”가 아니라,

- A가 누구와 연결되어 있고,
- 그 관계가 어떤 의미이며,
- 그 관계를 몇 단계 따라가면 무엇이 나오는가?

와 같은 것을 다룰 때 장점이 있다.

### 좋은 경우

- 데이터에 명확한 엔티티와 엔티티 간 관계가 존재한다.
  - 예: 의약품-성분-부작용 관계, 인물-조직-사건 관계
- 질문이 여러 엔티티를 거쳐가야 답할 수 있다. 즉 질문의 답이 한 문장 안에 바로 있지 않고, 여러 개체를 연결해서 찾아야 하는 경우이다.
  - 예: "A의 공급업체가 납품하는 다른 고객사는?"
- 관계의 방향성이나 속성이 중요하다.
  - 예: "부하직원이 아니라 결재권자가 누구인가"
    -   대리-[보고한다]->부장 
    -   부장-[관리한다]->대리  
    -   위 의 경우 방향성이 없으면 누가 누구에게 보고하고 관리하는지를 알 수 없으므로 결제권자인지 알 수 없다.
- 전체 그래프에 대한 요약이나 집계가 필요하다.
  - 예: "관객들이 가장 많이 본 영화는 무엇인가?"

### 좋지 않은 경우

- 데이터가 비정형 텍스트 위주이고 엔티티 추출이 어렵다.
  - 예: 감성 분석 대상 리뷰, 자유 형식 설문 응답
- 질문이 단순히 "내용 검색"에 해당한다.
  - 예: "환불 정책이 어떻게 되나요?" (FAQ 유형)
- 데이터의 관계 구조가 단순하거나 없다.
- 도메인 지식 없이 스키마를 설계하기 어려운 경우.
